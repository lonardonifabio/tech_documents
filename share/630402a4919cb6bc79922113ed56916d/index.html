SubGCache: Accelerating Graph-based RAG with Subgraph-level KV Cache

Authors: Qiuyu Zhu, Liang Zhang, Qianxiong Xu, Cheng Long, Jie Zhang

This research paper introduces SubGCache, a novel approach to accelerate graph-based retrieval-augmented generation (RAG) systems for large language models (LLMs). The core observation is that different queries often retrieve similar subgraphs, leading to redundant computation. SubGCache addresses this by clustering queries based on subgraph embeddings. For each cluster, a representative subgraph is identified, and its key-value (KV) cache is pre-computed. When a query with a matching subgraph is encountered, SubGCache reuses this pre-computed cache, eliminating the need for repeated KV tensor computations. This significantly reduces inference latency. Experiments conducted on two new datasets, utilizing multiple LLM backbones and graph-based RAG frameworks, consistently demonstrate a reduction in time-to-first-token (TTFT) ‚Äì achieving up to 6.68x. The system maintains comparable, and in some cases improved, generation quality. The proposed method offers a practical solution for optimizing the performance of RAG systems by intelligently reusing computations based on subgraph similarity, making it a valuable contribution to the field of efficient LLM inference.

Key Concepts:

  • Graph-based retrieval-augmented generation (RAG) utilizes graph retrieval to enhance LLMs.
  • SubGCache aims to reduce inference latency by reusing computation across similar subgraph prompts.
  • The system clusters queries based on subgraph embeddings and pre-computes a KV cache for each cluster's representative subgraph.
  • Representative subgraph construction: Merging subgraphs to preserve relational context.
  • Subgraph-level cache reuse: Efficient KV cache computation and reuse within clusters.
  • Graph Embedding via Pretrained GNN: Encoding subgraphs for similarity comparison.
  • Minimal processing overhead during clustering.
  • Low cluster processing time across configurations.
  • SubGCache's clustering stage introduces modest overhead.
  • OAG dataset incurs higher processing time due to larger graph size.
